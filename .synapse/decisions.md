# decisiones.md - Registro de Decisiones Técnicas

| ID | Decisión Técnica | La Razón (The Why) | Estado |
|:---|:---|:---|:---|
| D-01 | **Uso de Python para parsear DOCX** | No se disponía de herramientas nativas para visualizar DOCX; se programó un script en python utilizando zipfile y xml.etree.ElementTree para extraer el contenido a markdown. | 🔒 LOCKED |
| D-02 | **Next.js como Framework** | Next.js permite desarrollar frontend interactivo (Agenda en calendario, buscador de clientes) y backend (API routes, base de datos con Prisma, envío de correos/notificaciones) en un solo proyecto, facilitando el despliegue en un VPS propio. | 🔒 LOCKED |
| D-03 | **VPS Debian DonWeb + PostgreSQL** | El cliente posee un VPS de DonWeb con Debian. Usaremos PostgreSQL para producción por ser robusto y nativo en el servidor, manteniendo SQLite en desarrollo local para agilidad. | 🔒 LOCKED |
| D-04 | **Autenticación en Panel de Control** | Se confirma la necesidad de un login seguro para evitar acceso no autorizado a los datos de los clientes en `/admin`. | 🔒 LOCKED |
| D-05 | **WhatsApp Automatizado con whatsapp-web.js** | Se prefiere el envío automatizado a través de whatsapp-web.js (Puppeteer) en lugar de links manuales de wa.me. El servidor en Debian VPS ejecutará el navegador headless y mostrará el código QR en `/admin/configuracion` para la vinculación. | 🔒 LOCKED |
| D-06 | **Notificación de Faltazos vía SMTP** | Se seleccionó Nodemailer configurado con variables de entorno SMTP para enviar correos transaccionales con plantilla HTML premium cuando se registra inasistencia (`NO_ASISTIO`). | 🔒 LOCKED |
| D-07 | **Buscador predictivo (Autocomplete) del lado del Cliente** | Se optó por cargar la lista de clientes completa en memoria al abrir el modal de agendado en `/admin/agenda` para realizar la búsqueda y autocompletado en el frontend, garantizando respuesta instantánea y simplicidad de diseño. | 🔒 LOCKED |
| D-08 | **Rediseño de Modales Administrativos por Flexbox** | Rediseñamos el modal de la Ficha Digital del Cliente dividiéndolo en secciones estáticas (cabecera, pestañas) y un cuerpo scrollable independiente (`overflow-y: auto`). Esto fija el botón de cierre y pestañas arriba y evita scroll innecesario. | 🔒 LOCKED |
| D-09 | **Línea de Tiempo con cálculo local reactivo** | Calculamos la posición del indicador de hora actual en base a la diferencia en minutos entre la hora del día y la hora de inicio laboral (`config.work_start`), actualizándolo cada 1 minuto mediante hooks locales de react. | 🔒 LOCKED |
| D-10 | **Tematización Local por Variables CSS** | Redefinimos las variables de color globales (ej: `--bg-primary`, `--text-primary`) localmente dentro de la clase `.container` en `page.module.css`. Esto permite que las páginas de reserva públicas hereden el esquema de colores de la web oficial (`#f0ede6` crema y `#7a1f1e` bordó) sin afectar la estética oscura del panel administrativo (`/admin`). | 🔒 LOCKED |
| D-11 | **Validación estricta de unicidad en base de datos** | Aplicamos restricciones de DNI, correo y WhatsApp únicos en la capa de API para evitar fichas duplicadas del mismo cliente, validando también las modificaciones sobre perfiles existentes. | 🔒 LOCKED |
| D-12 | **Calendarización Mensual por Grilla Dinámica** | Diseñamos un renderizador mensual dinámico en el admin calculando los días de relleno del mes anterior y siguiente para mapear el calendario completo sin utilizar dependencias pesadas como FullCalendar. | 🔒 LOCKED |
| D-13 | **Consolidación de Crons Automáticos** | Integramos la lógica de recordatorio diario por correo a los 75 días dentro del cron de recordatorios de WhatsApp de las 10:00 AM para evitar crear múltiples tareas de consulta a la base de datos de manera paralela. | 🔒 LOCKED |
| D-14 | **Concatenación Transparente de Nombre/Apellido** | Decidimos mantener el esquema de base de datos intacto (campo `nombreCompleto` String) para evitar migraciones destructivas. En su lugar, el frontend divide el string al cargarse y concatena los campos `nombre` y `apellido` transparentemente antes de enviar los datos al backend. | 🔒 LOCKED |
| D-15 | **Normalización Estricta de WhatsApp en Backend** | Creamos la función `normalizeWhatsApp` en `src/lib/whatsapp.js` para estandarizar números al formato numérico limpio `549xxxxxxxxxx` antes de guardarlos o validar duplicidad. Esto previene registros repetidos por formatos variados (espacios, guiones, ceros o prefijo 15). | 🔒 LOCKED |
| D-16 | **Vista Imprimible Limpia sin Dependencias** | Implementamos una ruta específica `/admin/agenda/imprimir` que renderiza una tabla A4 minimalista en blanco y negro aplicando reglas `@media print` para ocultar la botonera y las barras laterales nativas del navegador, disparando automáticamente el cuadro de diálogo del sistema (`window.print()`). | 🔒 LOCKED |
| D-17 | **Paso de Observaciones al Registro del Cliente** | Decidimos asociar las observaciones de los turnos directamente a la tabla `Cliente` (`cliente.observaciones`) en lugar del turno individual, facilitando que las anotaciones se conserven históricamente y se visualicen/modifiquen en cascada desde cualquier turno. | 🔒 LOCKED |
| D-18 | **Guardado de Zona Extra 'Otros' como Objeto JSON** | Guardamos la zona personalizada "Otros" como un objeto `{ id: 'otros', nombre: 'Otros: [texto]', precio: 0, duracion: 0 }` dentro del array serializado `zonas` del turno, permitiendo compatibilidad nativa con reportes y facturación sin alterar la estructura del esquema. | 🔒 LOCKED |
